## The 5 findings that matter most (ranked by severity)

### 1. "Existing IR unchanged" is false — 5 underscore fields break the claim *[Reviewer 3]*

`_node_id_map`, `_mode1_eligible`, `_corpus_source_node_id`, `_original_name`, `_composition` are load-bearing, consumed at 14+ renderer sites, and have no representation in the markup grammar. The MVP's "additive" claim assumes the bridge is thin; it isn't. `_corpus_source_node_id` is a raw integer, directly violating invariant 7 ("IR never holds raw values"). The compose-path vs round-trip-path distinction for `_node_id_map` alone is a significant design decision the plan doesn't acknowledge.

**Blocks:** MVP Day 6 (bridge) + all of Stage 4.
**What to do:** either extend the grammar to represent these fields OR declare them bridge-time-computed with explicit reconstruction logic. Not both, not neither.

### 2. "One grammar" claim is false — there are at least two grammar modes *[Reviewers 1, 3]*

Extract path emits raw values pre-clustering (Stage 1). Synthesis path requires token-only grammar-constrained decoding (Stage 4). These are two different constraint rules on the same syntax. Also: ADR-001 capability table and proposed KDL grammar are two constraint sources that must stay in sync. The "single source of truth" from ADR-001 is compromised the moment the markup grammar exists as a separate artifact.

**Blocks:** Stage 3 (synthetic tokens) depends on reconciling these. Invariant 7 as written is violated Day 7 of MVP.
**What to do:** explicit grammar modes (extract permits raw literals, synthesis prohibits), OR a clean derivation path from capability table → grammar.

### 3. Day 6 IR bridge is 5-8 days, not 1 — MVP is 20-35% probable as scoped *[Reviewer 2]*

The bridge must invent serialization for: gradient `handlePositions` (float 2D arrays), override trees with Figma path syntax, vector paths (324+ char strings), synthetic negative `_node_id_map` entries, Mode-1 flags. Plus: `parse(emit(ir)) == ir` has no equality function (float precision, ephemeral keys, synthetic eids — writing correct equality is 1-2 days itself). Plus: no KDL parser in Python ecosystem — 3-4 days for hand-written recursive-descent for v2 (type annotations, multiline, escapes).

**Blocks:** MVP timeline.
**What to do:** Reviewer 2's smaller MVP (5 days): emitter+parser for 8 field types in `map_node_to_element` output, `parse(emit(spec)) == spec` on 5 screens. No Figma round-trip — just grammar completeness proof.

### 4. Zero usability evidence for the markup as a designer-facing surface *[Reviewer 4]*

Users see `@eid`, `#eid`, `->`, `&`, `{tokens}`, `(provenance)`, `#[trailers]`, 7 edit verbs, `use … as …`, wildcards, dot-namespaces. 18 research threads on correctness. **Zero threads on comprehension.** All cited precedents (KDL, Elm, MLIR, React, Sass, Svelte 5) are engineer-facing. Users are designers.

**Blocks:** product-market fit assumption.
**What to do:** before Day 1 of MVP, hand-write 5 markup files, put in front of 2 non-engineer designers, observe. Either they can edit or they can't. The "LLM-friendly therefore human-friendly" assumption is unverified.

### 5. No buyer, no distribution, no pricing — it's a research paper *[Reviewer 5]*

Architecture doesn't name a target persona. No install-and-run product. Primary monetizable feature (multi-target native renderers) explicitly deferred beyond v0.3. MVP is internal round-trip test — no user ever sees it. Competitive window of 12-18 months acknowledged but no go-to-market plan.

**Blocks:** strategic viability.
**What to do:** Reviewer 5 proposed collapsing into a Figma plugin that ships in 6 weeks, with the KDL IR as invisible infrastructure. "Figma in, code out, verified." Architecture unchanged — sequencing inverted.

## Secondary issues — worth fixing but not MVP-blocking

- **Axis-polymorphism ≠ density** (Reviewer 1). Distinct concepts, conflated. CRAG scoring and Gate 2 need operational density definitions.
- **Verifier emits to RenderReport — that's a breaking schema change** (Reviewer 3). RenderReport is frozen dataclass. Adding `proposed_edits` field touches boundary.py, verify_figma.py, every consumer. Architecture calls "additive"; it's not.
- **Edit grammar doesn't cover Tier 1–4 token operations** (Reviewer 1). `set/append/insert/delete/move/swap/replace` — no `define-token`, `bind`, `unbind`. "One grammar" compromised for action-taxonomy operations.
- **`_UNIVERSAL_MODE3_TOKENS` has duplicate keys in compose.py** (Reviewer 3). Lines 444+446 and 445+447 shadow each other. Production vocabulary doesn't match any documentation. Stage 3 would inherit the bug.
- **Pattern Language "reframe" is a straw man** (Reviewer 1). Original never claimed levels were markup-structural. Reframe drops actual content (decision tree as persisted data) without acknowledging.
- **Multi-target "just schema extension" understates target-specific behavioral divergence** (Reviewer 1). Figma autolayout vs CSS flexbox vs SwiftUI sizing don't reduce to `prop_map` enums.

## What the reviewers agree on

- **The architecture is genuinely good.** Axis-polymorphic markup, shared grammar, CRAG cascade, verifier-as-agent — all defensible, research-validated. (Reviewers 1, 4, 5)
- **The invariants are real.** 204/204 parity, 1,950+ tests, 8 ADRs — these constraints exist and matter. (All reviewers)
- **The MVP scope is aspirational at best.** The concrete work-per-day is underestimated everywhere. (Reviewers 2, 3)
- **The product surface is underspecified.** What a user does with this thing on Day 1 is not answered. (Reviewers 4, 5)
- **Competitive window matters.** Builder.io and Figma moves in the next 6 months have been modeled honestly; the response hasn't. (Reviewers 4, 5)

---

## What the reviewers disagree on

- **Reviewer 5 says: pivot to Figma plugin, ship in 6 weeks.** Invisible infrastructure underneath.
- **Reviewer 2 says: smaller MVP (5 days) proves grammar completeness without Figma round-trip.** Then full round-trip is a separate sprint.
- **Reviewer 4 says: before ANY implementation, run a usability test on hand-authored markup.** Validate designer comprehension, not engineer aesthetics.
- **Reviewers 1 and 3 say: resolve the architectural inconsistencies first.** Underscore fields, raw-value grammar modes, capability-table/grammar sync — these must be settled before MVP begins.

These are four distinct recommendations about what to do. They're not mutually exclusive, but they're not the same either.
