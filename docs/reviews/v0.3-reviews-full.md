# v0.3 Architecture — Review Reports (Full)

5 independent sonnet-based reviewers, dispatched 2026-04-18 in parallel, reviewing the three v0.3 planning documents. Each reviewer was given a distinct angle and told to be critical, not flatter. The synthesis is in `docs/reviews/v0.3-review-synthesis.md`. Full reports preserved here for reference.

---

## Reviewer 1 — Architectural consistency

**Angle:** detect internal contradictions; audit if the 10 load-bearing decisions cohere.

### Top 3 architectural inconsistencies

**1. Axis-polymorphism and density-per-node are conflated.**

Architecture §1.1 defines axis-polymorphism as "any subset of axes can be populated." §1.2 declares this is what "density" means. They're distinct: axis-polymorphism is which axes have values; density is how much is filled within each axis. A node can be spatially dense (pixel-perfect sizing) but visually sparse (only a fill, no stroke). The architecture describes density only as axis-presence. This matters for CRAG scoring and Gate 2 — neither has an operational definition of density.

**2. The "one grammar" claim is false.**

Invariant 7: "IR never holds raw values; synthetic tokens fill gaps." But the clustering pass is Stage 3, AFTER the markup emitter (Stage 1). Extracted IR legitimately holds raw hex values before clustering. Grammar-constrained decoding for synthesis prohibits raw literals; extraction path must permit them. There are AT LEAST two operating modes of the grammar — the architecture never acknowledges this fork.

**3. CRAG SYNTHESIS vs "quality ceiling → regenerate" vs Best-of-N don't reconcile.**

Below Gate 1+2, system regenerates. In SYNTHESIS mode, regeneration = re-running L1→L2→L3 Pattern Language descent (expensive). Best-of-N (N=3-5) also runs L1→L2→L3 N times. The architecture advocates both but never says whether Best-of-N IS the L1-branching optimization, or whether they're separate.

### Top 3 glossed-over complexities

**1. Multi-target as "just a schema extension" understates target-specific behavioral divergence.** Figma autolayout vs CSS flexbox vs SwiftUI `.frame(maxWidth: .infinity)` don't reduce to `prop_map` enums. Whether target-specific capability gating is needed is not addressed.

**2. Edit grammar scope untested against action taxonomy.** Tier 1–4 token-system operations (T2.2 split-token, T2.3 create-alias) need grammar verbs like `define-token`, `create-alias` that don't exist in the 7-verb set. "One grammar" claim is compromised for those ops.

**3. Provenance annotations and the existing IR.** MVP says "existing IR unchanged." Architecture §2.6 introduces four provenance kinds as queryable. Current IR has only `_corpus_source_node_id` (a single int). These aren't optional decorators — they're structural requirements for the verifier's repair-target queries.

### Research-to-architecture mismatches

**Pattern Language reframe is a straw man.** The architecture claims to "reframe" Pattern Language from "markup structural" to "pipeline optimization." But `t5-pattern-language.md` never claimed levels were markup-structural — it explicitly described them as a compositional process. The architecture rejects a claim the original didn't make, then drops genuine content (decision tree as first-class persisted data) without acknowledging the loss.

### Verdict

**Moderate inconsistencies, not fatal.** Three are load-bearing for implementation:
1. Raw-value grammar window (hits Week 1 of MVP)
2. Tier 1–4 token operations not in 7-verb set (breaks "one grammar")
3. Axis vs density conflation (creates ambiguous CRAG scoring)

### Changes recommended

1. Clarify raw-value grammar status — two modes explicitly, or one mode with validation-rule variants.
2. Audit edit verbs against Tier 1–4 — add verbs or formally remove from "shared grammar" claim.
3. Define density within an axis with a formula.
4. Multi-target capability table — specify per-target or single.

## Reviewer 2 — MVP + implementation risk

**Angle:** is the 2-week MVP realistic? What breaks first?

### The single most underestimated task

**Day 6 (IR bridge) is 5-8 days, not 1.**

The bridge couples the new markup module to existing `ir.py` (1,436 lines). CompositionSpec for a 376-node screen carries:
- `override_tree` with semicolon-delimited Figma path syntax
- Gradient fills with `handlePositions` (float 2D arrays) alongside `gradientTransform` (Plugin API) — not equivalent, can't drop one
- 27,811 INSTANCE elements with `_mode1_eligible` flags
- Vector elements with 324+ char SVG path strings
- Internal bookkeeping: `_node_id_map`, `_mode1_eligible`, `_original_name`

Each compound type requires a serialization invention. `handlePositions` alone — do you embed as KDL property list, child node, base64? — each choice has round-trip and constrained-decoding implications.

Realistic Day 6: 1 day per-type format decisions, 2 days emit impl, 2 days parse-back, 1-2 days edge cases. **6-7 days minimum.**

### Top 3 risks missed

**Risk A: "No regressions" gate only runs Day 14.** 10 days of silent risk accrual. A new import in `dd/__init__.py` with a side-effect breaks the pipeline in a way not surfaced until bridge runs. Should run sweep Day 7.

**Risk B: `parse(emit(ir)) == ir` has no equality function spec.** IR contains floats like `1.0053191048296253`, positional eids, internal bookkeeping, synthetic `_node_id_map` entries. Writing correct `assert_spec_equal` with float-within-epsilon, eid stability handling, ephemeral-key exclusion is 1-2 days itself. Zero budget in plan.

**Risk C: No KDL parser in Python ecosystem.** `kdl-py` appears unmaintained. `tree-sitter-kdl` for v2 (finalized 2024) may not exist. Hand-written recursive-descent is realistically 3-4 days for KDL v2's type annotations, multiline strings, escape sequences.

### Probability of 100% round-trip parity on 20 screens in 2 weeks: **20-35%**

If re-scoped to "IR fields used by `generate_figma_script`" (ignoring fields the renderer doesn't consume like raw `handlePositions`): **50-60%**.

### Plan changes suggested

1. **Split emitter from bridge.** Days 4-5 emit `IR_element → KDL`, print real IR for 5 screens BEFORE writing emitter code to design format.
2. **Equality function on Day 2, not Day 10.** Surface assumptions early; 1.5-day budget.
3. **Regression sweep on Day 7.** 30 min cost, saves 3 days of debugging.
4. **Drop constrained-decoding smoke test.** Premature bridge to Stage 4; use as buffer.

### Smaller MVP proposal (5 days)

Write emitter+parser for the 8 IR field types in `map_node_to_element` output: solid/gradient fills, strokes, effects, layout, props, children, instance. Verify `parse(emit(spec)) == spec` on 5 screens. **No Figma render. No Plugin API. No sweep.** Proves grammar completeness in 5 days. Full round-trip becomes a separate sprint.

## Reviewer 3 — Existing-codebase alignment

**Angle:** verify the "additive + flag-gated" claim against actual code.

### Places "X is unchanged" is wrong

**The existing IR carries 5 underscore fields not documented in architecture:**

1. **`_node_id_map`** — read at 14 sites in `figma.py` (173, 910, 1000, 1175, 1207, 1228, 1461, 1481, 1553, 1639, 2401). Compose path sets it to `{}` at compose.py:316; round-trip path fills it at ir.py:1384. The markup emitter must distinguish real DB ids from synthetic negative ids.
2. **`_mode1_eligible`** — set ir.py:342, consumed verify_figma.py:182. Decides if Mode-1 degradation is a real error. Missing from markup = false type_substitution errors.
3. **`_corpus_source_node_id`** — compose.py:160, corpus_retrieval.py:434. Consumed by build_template_visuals to prefer DB visuals over template defaults. This is a raw integer, not a token ref — **violates invariant 7**.
4. **`_original_name`** — ir.py:1271, 1378, corpus_retrieval.py:454; consumed figma.py:1044/1080/1191/1597 for placeholder naming.
5. **`_composition`** — set by build_template_visuals.

**`_UNIVERSAL_MODE3_TOKENS` has duplicate keys.** `space.generic.padding_x` appears at compose.py:444 AND 446 with different values; `space.generic.padding_y` at 447 overrides 445. Python dict takes last-write — first declarations are silently shadowed. Production token vocabulary doesn't match any documentation.

### ADR violations

**ADR-001 (capability-gated emission).** ADR-001 says capability table IS the constrained-decoding grammar for synthetic IR. KDL grammar encodes a different structure (axis-polymorphic nodes with value forms). These are two different constraint languages. If they drift, ADR-001's "single source of truth" is broken. Architecture says "grammar extensions must register capabilities" but doesn't explain the mapping.

**ADR-007 (verifier-as-agent channel).** RenderReport is a frozen dataclass with KIND_* errors. Adding `proposed_edits: list[MarkupEdit]` is **not additive** — it's a breaking schema change touching boundary.py, verify_figma.py, every RenderReport consumer, sweep runner. Architecture calls this additive; it isn't.

### Integration points underestimated

1. **`_node_id_map` compose-path vs round-trip-path ambiguity.** Emitter must know which path produced the IR — synthetic negatives can regenerate; real DB ids must survive. MVP Day 6 says "make sure `_node_id_map` survives" but doesn't address compose-path synthetic case.
2. **v0.2 `element["visual"]` fast path invisible to emitter.** build_template_visuals:984-1000 bypasses template lookup when `element["visual"]` is present. Grammar has no value form for a pre-normalized visual dict. Emitter must flatten and re-detect — not trivial.
3. **CorpusRetrievalProvider.supports() needs live DB.** Bridge module needs DB injection. MVP doesn't address.

### Is "additive + flag-gated" achievable? **Partially.**

Parser and emitter can be additive. **Bridge cannot** — it must produce a CompositionSpec dict the existing renderer consumes. That means populating `_node_id_map`, handling `_mode1_eligible`, deciding `_corpus_source_node_id`. Bridge is a third producer of an existing contract.

### Refactors plan must call out

1. Formalize underscore-field contract — either in grammar or documented as bridge-time computed.
2. Fix `_UNIVERSAL_MODE3_TOKENS` duplicate keys before Stage 3 uses it as baseline.
3. Declare `RenderReport` schema extension explicitly as breaking change (Stage 5).
4. Document compose-path vs round-trip-path distinction for `_node_id_map` — either gain a `_source_path` field or inspect map sign.
5. Clarify parity is measured on extracted screens, not compose-path screens — different measurements apply.

## Reviewer 4 — Red team

**Angle:** find the scenario that kills the plan.

### Most likely to kill: "The markup's too complicated, and you have no evidence it isn't"

User opens emitter output for a 400-node screen. 1,200 lines of indented KDL. They want to edit the "Sign In" button at `@header/nav/auth-controls@0/button@1`. Can't find it. Guess wildcard. Nothing changes. Close file. Back to Figma.

**The plan has zero usability gate.** 18 research threads on correctness, zero on comprehension. Cites KDL/Elm/MLIR/CSS Shadow Parts/React/Svelte 5/Sass — all engineer-facing tools. Users are designers.

**What should happen:** before 2 weeks of parser/emitter work, hand-write 5 representative markup files and put in front of 2 non-engineers with no explanation. Either they can find elements they want to edit, or they can't. If they can't, every assumption about "LLM-friendly therefore human-friendly" is wrong. These are not the same property.

### Five hidden assumptions

1. **42K Dank nodes generalize.** Deferred to v0.4; CRAG τ_high calibrated on Dank donors. Fintech user's retrieval scores are wrong.
2. **Constrained decoding over KDL+annotations is tractable.** MVP smoke test is day 11-12 of a 14-day sprint. KDL with provenance trailers + slash-dash + dot-namespace + value-form polymorphism is not a simple grammar. Mitigation in risk table is "if misalignment, simplify grammar or pick different library" — that's a full redesign event.
3. **Round-trip parity proves synthesis quality.** MVP is round-trip; users buy synthesis. Gate is fidelity 0.728 → 0.75 (2.2% lift). Actual user question ("does this produce designs a designer wants?") never operationalized. Taste model deferred to v0.4.
4. **Verifier doesn't hallucinate registry entries.** Constrained decoding knows token vocab; doesn't know live eid space unless full IR re-fed every repair iteration. No cost model.
5. **2-week MVP independent of IR discovery.** IR has 72 columns, multiple sessions of quirks (supplement extraction, coordinate divergence, gradient format bugs, text layout quirks — all in MEMORY.md). `$extensions` bucket mitigation immediately compromises "markup is the single grammar."

### Comparative marketing problem

- **vs Figma First Draft + Code Connect:** they have zero context-switching; we require CLI + markup editor. "Our fidelity is higher" vs "I only care about React and fidelity is close enough."
- **vs v0 + Mitosis:** they're prompt→React→many-targets from scratch. Our bidirectional is a tax, not a benefit, for code-first teams.
- **vs Cursor + design-system-MCP:** developers don't need Figma in the loop; they need token-referenced code. Our Figma bidirectionality is the wrong wedge.

### Prediction (6 months): **(b) MVP slipped, competitors shipped**

Week 1-2: parser+emitter produces. Day 7 surfaces 2-3 IR fields that don't map cleanly (gradient handlePositions, override trees, vector paths). `$extensions` bucket handles but "single grammar" claim qualified. +3 days. Day 11 constrained decoding hits KDL/GBNF conflict. Choosing between simplified grammar (breaks research-validated decisions) vs switching library (Python bindings + new dep) takes a week. Stage 1 ships in 3 weeks. Stage 4 takes 4 weeks — τ threshold calibration requires more labeled data than expected. At 6 months: working parser, CRAG pipeline on 12 prompts, fidelity 0.77, **no external users, no portability beyond Dank, a competitor shipped the headline claim.**

The thing not predicted: MVP slip came from IR having more surface area than architecture assumed. Invariant "204/204 parity is load-bearing" forced never compromising on IR completeness. KDL couldn't express everything cleanly. Grammar grew. Constrained decoder had to be tested against grown grammar. Every other stage waited.

## Reviewer 5 — Strategic positioning

**Angle:** is this a product someone will buy?

### Primary user and value prop

Architecture doesn't name a buyer. Only person with immediate value: design-systems engineer at a product company with 20+ engineers, established design system, active iOS/Android/web surface. Think Shopify/Airbnb/mid-fintech.

Value prop: *"Extract your existing Figma design into a typed IR, edit it once, push verified output to Figma + React + SwiftUI from a single source of truth."*

**That sentence is defensible if the multi-target renderers actually ship. Until they do — architecture explicitly defers React and SwiftUI production renderers beyond v0.3 — the value prop is a spec, not a product.**

### vs Figma + Cursor + MCP positioning — three real differentiators

1. **Structural round-trip, not lossy snapshot.** MCP trades fidelity for context. Can't confirm token binding survived code edit. v0.3 can.
2. **Grammar-constrained generation prevents hallucination.** Cursor + MCP generates non-system colors silently.
3. **Corpus as starting point.** CRAG retrieves and edits; MCP stack generates from scratch every prompt.

Defensible but narrow: requires user who has Figma design system with token bindings, cares about parity, builds for >1 platform. **Small market.**

### Three things customers don't care about

1. **204/204 round-trip parity as marketing.** Customers buy "didn't drift," not "passes parity." Mechanism vs value.
2. **KDL as the markup language.** Users care about editor tooling, not grammar theory. KDL has minimal editor support today.
3. **Axis-polymorphic 5-axis model.** Solves internal pipeline coherence. Invisible to user.

### Three things customers want that architecture doesn't address

1. **Install-and-run product.** MVP is a Python module requiring extracted corpus, DB, WebSocket plugin, CLI. No distribution, no installer, no UI, no auth, no team model.
2. **Real token+component mapping for 200-component design systems.** `prop_map` example shows one clean case. Real systems are maintained by non-engineers.
3. **Faster than the existing workflow.** Verifier loop takes 45-90 seconds per iteration. Designer clicks in Figma in 5 seconds. Speed is missing from the architecture.

### Would this get a check? **No.**

Architecture is impressive. Research is thorough. **But it's a research paper, not a product plan.** Missing:
- Target buyer with job title + quantified pain
- Distribution strategy
- Pricing hypothesis
- Primary monetizable feature (multi-target native) explicitly deferred
- MVP is internal round-trip test, no user sees it
- 12-18 month window acknowledged, no plan for who ships first

### The one pivot that would make this defensible

**Collapse into a Figma plugin that ships in 6 weeks.** Full v0.3 architecture is valuable — round-trip extractor, classified corpus, token pipeline are real assets. But customer-facing surface for 12 months shouldn't be KDL markup.

- Figma plugin
- Designer selects frame, clicks "Export to Code"
- Selects target (React / SwiftUI)
- Gets structurally verified, token-bound output in clipboard
- No markup authoring, no corpus pipeline visible
- KDL IR runs underneath, never exposed

Three effects: day-one experience for designer/frontend engineer without architecture comprehension; distribution channel obvious (Figma Plugin Store, free tier, paid for multi-target/team); competitive claim simple ("we verify before you get it"). Architecture is invisible infrastructure. Markup authoring becomes v0.4 power-user feature after plugin validates demand.

**Architecture isn't wrong. Sequencing is.** Build invisible infrastructure first, ship plugin second — ship plugin to users before infrastructure is fully complete. Discovery through usage, not architecture documents.
