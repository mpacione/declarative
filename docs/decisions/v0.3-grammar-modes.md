# v0.3 Decision: Grammar Modes — Raw vs Token-Only

**Status:** Proposed 2026-04-18.
**Related:** Priority 1B per `docs/continuation-v0.3-next-session.md` §4; resolves reviewer 1 + reviewer 3 "one grammar" framing concern.

## 1. Problem

Reviewers 1 and 3 flagged that the v0.3 architecture claims **one grammar** but operates in two validation regimes:

- **Extract path** — reads raw pixel values from Figma, emits IR with raw hex / pixel numbers / strings. Clustering hasn't happened yet.
- **Synthesis path** — constrained decoding requires token refs (`{color.brand.primary}`) for clusterable axes; raw values are invariant-violations.

ADR-001's capability table is a third constraint source (per-backend emission eligibility), adding to the "single grammar" ambiguity.

## 2. Reframing after Priority 0

With conservative Priority 0 (dd markup = serde over dict IR), the grammar (syntax) and validation (semantics) are separable concerns:

- **Syntax** (what dd markup *can* represent) — one grammar. Strings, numbers, braces, token refs all parseable.
- **Validation mode** (what a *specific document* is *allowed* to contain) — two modes.

This is standard in language design. Compare: TypeScript the language is one grammar; `strict: true` vs `strict: false` are two validation modes.

## 3. Decision

**One syntactic grammar, three validation modes.**

### Mode E — Extract-time validation (raw permitted)

Applied to IR produced by `dd/ir.py::generate_ir` directly from DB rows.

- Raw hex (`#F6F6F6`), raw numbers (`16`), raw strings (`"Inter"`) all allowed
- Token refs (`{color.brand.primary}`) also allowed (if DB has the binding)
- Mixed values on one axis are allowed (different screens, different clustering maturity)
- Fails only on: missing required fields, type-mismatch (string where number expected), structural invariants (unknown eid references)

### Mode S — Synthesis-time validation (token-only on clusterable axes)

Applied to IR emitted by the LLM during synthesis / composition / edit.

- On **clusterable axes** (colors, spacing, radii, typography), raw values are rejected. Must use token refs.
- On **non-clusterable axes** (position coordinates, eids, strings that are content like user-facing text), raw values are required (there's no cluster for "button label text").
- Token refs must resolve in the cascade (`TokenCascade.resolve` succeeds) — else `KIND_TOKEN_UNRESOLVED`.

### Mode R — Render-time validation (capability-gated)

Applied when the renderer is about to emit a property.

- ADR-001 capability table per backend (figma / react / swift / flutter).
- If the IR has a property the backend can't emit, the capability gate drops it with a structured warning.
- Not about raw-vs-token — about "does this backend support `shadow.inner=...`".

**The three modes compose.** An IR that passes Mode E (extract-valid) can be fed through clustering to produce an IR that passes Mode S (synthesis-valid). Render (Mode R) runs on either.

## 4. Invariant 7 reworded

Current phrasing (per the aspirational architecture): "IR never holds raw values."

Actual behavior: IR sometimes holds raw values (extract path, pre-clustering).

New phrasing (to be written into `architecture-v0.3.md` §9):

> **Invariant 7.** Token refs are the canonical form for clusterable-axis values in synthesis output and LLM-facing IR. Extract-produced IR may hold raw values on any axis; running it through clustering (`dd/cluster*`) promotes raws to refs where a cluster binding exists. Renderers must accept either form and resolve via `TokenCascade` as needed.

And adding a new invariant that honest surfaces the mode distinction:

> **Invariant 7a.** The dd-markup grammar is one syntax. Validation operates in three modes: Extract (raw permitted), Synthesis (token-only on clusterable axes), Render (backend-capability-gated). Any given IR document is valid under one or more of these modes; documents from different sources must not be assumed to be valid under all modes.

## 5. Where clustering fits

`dd/cluster.py` + `dd/cluster_colors.py` + `dd/cluster_spacing.py` + `dd/cluster_typography.py` + `dd/cluster_variants.py` are the promotion step. They read raw values from the DB corpus, group them into clusters, assign canonical token names, and write bindings into `variant_token_binding` + similar tables.

**After clustering has run on the corpus:**
- The DB has binding rows
- `dd/ir.py::generate_ir` + binding-overlay logic produces IR with `{token.name}` refs where bindings exist
- Clusterable-axis raw values survive only where no binding exists (rare by design; the cluster is supposed to cover the axis)

**Before clustering has run (bootstrap):**
- All clusterable-axis values in IR are raw
- Synthesis can still run in SYNTHESIS mode because of the **synthetic tokens** layer (`_UNIVERSAL_MODE3_TOKENS` in `dd/compose.py` and friends) — universal defaults fill the token namespace so Mode-S validation has something to resolve against

This is **cold-start-safe**: a fresh project with no clustering gets usable token refs via universals. A project with mature clustering gets project-specific tokens.

## 6. Implications for the dd-markup parser

The parser built in the Priority 0 probe (`dd/markup.py`) handles Mode E (raw-permitted) trivially — any string/number/bool parses. It does NOT enforce Mode S — that's a separate validation pass layered on the parsed AST.

**Follow-up:** implement `dd.markup.validate(ir, mode="S")` → `list[StructuredError]`. Uses the capability table + cascade to check every clusterable-axis value is a resolvable token ref.

## 7. What this does not resolve

- The **definition of "clusterable"** is currently implicit across the cluster_*.py modules. A formal capability-table entry per property (already ADR-001's shape) would make Mode S validation mechanical. Listed as a follow-up.
- "Content strings" vs "identifier strings" currently rely on context. A typed schema for element fields (`text: Content`, `eid: Identifier`) would make this explicit — but that's schema work, not mode work.
