# v0.3 Decision: Parity-gated branching + CI strategy

**Status:** Proposed 2026-04-18.
**Related:** Priority 4 per `docs/continuation-v0.3-next-session.md` §4; invariant derived from ADR-007 (unified verification channel).

## 1. Problem

204/204 round-trip parity on Dank Experimental is our foundational invariant. Silent parity regressions during v0.3 work are worse than v0.3 work that doesn't ship. We need an explicit branching + CI strategy that makes parity loss impossible to land, not a "we'll check at the end" approach.

## 2. Decision

Three-tier verification, matching three natural work cadences:

### Tier 1 — unit + structural (every commit)

- `pytest tests/` must pass.
- `pytest tests/test_markup_roundtrip.py` (dict-level 204/204) must pass.
- Runtime: ~30s.
- Enforced via: developer habit. Optional pre-commit hook.

### Tier 2 — script-parity (every PR to v0.3 integration branch)

- Script-identity sweep: for each of 204 screens, `generate_figma_script(baseline_spec) == generate_figma_script(round_tripped_spec)` must hold byte-identically.
- Runtime: ~15s (no Figma bridge required).
- This is the proof we already built (`docs/decisions/v0.3-canonical-ir.md` §3.2). Promote it to a reusable test.
- Enforced via: PR gate (see §3 below).

### Tier 3 — pixel-parity sweep (every merge to `main` + nightly)

- Full `render_batch/sweep.py` with `DD_MARKUP_ROUNDTRIP=1` must yield `is_parity_true >= 202` (allowing up to 2 transient walk-failures per `memory/feedback_sweep_transient_timeouts.md`).
- Runtime: ~20 minutes. Requires Figma plugin + bridge running.
- Enforced via: manual merge gate + nightly cron.

## 3. Branch topology

- **`main`** — 204/204 parity (or 202/204 with transient forgiveness). No broken commits.
- **`v0.3-integration`** — long-lived, rebases from `main` weekly. Tier 2 required before merging feature branches in. Tier 3 run before merging to `main`.
- **Feature branches** — short-lived (<1 week), named `v0.3-<topic>` (e.g. `v0.3-dd-markup-probe`, `v0.3-edit-grammar-server`). Rebase against `v0.3-integration`, not `main`. Tier 1 enforced; Tier 2 optional during exploration.

## 4. Rollback protocol

If a commit on `main` drops parity below 202/204:

1. Revert the offending commit with `git revert` (NOT `git reset --hard`).
2. Open an issue tagged `parity-regression` with the failing-screen list + baseline/actual is_parity counts.
3. Re-run Tier 3 on the revert to confirm recovery.
4. Fix-forward on a feature branch; no loose commits on `main`.

If a nightly Tier 3 catches a regression that slipped through because the merging PR only ran Tier 2:

1. Same revert protocol.
2. Post-mortem: identify what the PR's Tier 2 missed that Tier 3 caught. This is a signal to strengthen Tier 2.

## 5. The belt-and-suspenders case for keeping Tier 3

Tier 2 (script byte-identity) is strictly stronger than Tier 3 (pixel parity via Figma sweep) at detecting markup-serde regressions. Any change that breaks Tier 2 will also break Tier 3, but Tier 2 runs 80x faster.

Why keep Tier 3 then? Four reasons:

1. Figma-side non-determinism we haven't characterized (bridge version drift, plugin cache state, font substitutions) can affect renders without affecting scripts. Tier 3 catches these.
2. When the dict IR schema changes (not just the serde), script-identity tells us "the round-trip preserved it" but not "the renderer still handles it correctly." Tier 3 catches renderer regressions from IR schema changes.
3. Nightly cadence means regressions get found within 24 hours, not when someone next runs a sweep manually.
4. ADR-007's unified verification channel already requires pixel parity as the ground truth for `is_parity`. Tier 3 IS that verification.

## 6. What this does not cover

- **Synthetic screen generation** has no known ground truth. Its validation is orthogonal (vision-LM scoring, per v0.2 methodology). This doc is about round-trip/edit work only.
- **Performance regression** is not part of the parity gate. If a change makes a sweep take 40 minutes instead of 20, that's noise unless it crosses a timeout; we'd want a separate perf track.
- **Cross-file Figma corpus expansion** (beyond Dank Experimental) is out of scope until the first parity baseline is extended.

## 7. Implementation checklist

- [ ] Promote the script-parity probe into `tests/test_script_parity.py` (from the ad-hoc script in Priority 0 decision record §3.2)
- [ ] Add a `scripts/full_sweep.sh` wrapper that sets `DD_MARKUP_ROUNDTRIP=1` and invokes `sweep.py --port $BRIDGE_PORT`
- [ ] Document the Figma plugin/bridge startup procedure in a pinned location (suggest `docs/operational/figma-bridge-setup.md`)
- [ ] If GitHub Actions is desired: Tier 1 on every push; Tier 2 on every PR; Tier 3 nightly via self-hosted runner with Figma desktop available
- [ ] Branch-protection rules on `main`: require Tier 2 green before merge

## 8. Consequences

- Feature work moves slightly slower (~15s Tier 2 hit per PR) — justified trade for zero silent regressions
- Nightly Tier 3 consumes Figma desktop for ~20 minutes — schedule off-hours
- Rollback protocol explicit; no arguing after a regression lands
- ADR-007's pixel-parity invariant becomes enforced by process, not just intent
