# v0.3 Decision: Canonical IR Question

> ### ⚠ FULLY SUPERSEDED 2026-04-18 (late-late session) — see `docs/decisions/v0.3-option-b-cutover.md`
>
> Both the original "conservative" stance AND the 2026-04-18 mid-session revision (below) are superseded. The current position is **Option B**: dd markup IS the canonical IR end-to-end; the Figma renderer walks the L3 AST directly and no dict-IR intermediate exists on the render path.
>
> Trigger: a Figma sweep attempt (`render_batch/sweep.py --via-markup`) produced 0/3 pixel parity on smoke-test screens. Root cause analysis surfaced that the Option A path required multiple compile-time side-channels (`$ext.nid`, proposed `$ext.spec_key`) to preserve dict-IR identity across the lowering boundary — scaffolding for a system that was always meant to be demolished. The delta from "finish Option A" to "adopt Option B" was one week of additional work, paid once against carrying the scaffolding through every subsequent stage.
>
> Full rationale + migration milestones: `docs/decisions/v0.3-option-b-cutover.md`.
>

**Status:** Resolved 2026-04-18.
**Position:** **Conservative** — dd markup is a lossless serde over the dict IR; the dict IR remains canonical in memory and on the render path.
**Revisit trigger:** drift pressure between markup and dict IR schemas becomes unacceptable, OR we need the aggressive refactor for a specific feature that can't be expressed via serde.

## 1. The question

> Is dd markup the canonical IR going forward, or is it a projection of the dict IR with a bidirectional bridge?

Raised 2026-04-18 (reviewer 3 + user's follow-up on dual-representation debt). The M×N problem (`memory/project_ir_purpose.md`) re-emerges if two representations coexist.

## 2. Two positions considered

**Aggressive (unify):** dd markup becomes the canonical in-memory and on-wire IR. Extract emits parsed-dd AST. Renderers walk parsed-dd AST. Registry/capabilities constrain the markup schema — single source of truth.

**Conservative (bridge):** dd markup is a serialization + edit format layered over the existing dict IR. Bijective serde between them. Faster to ship; preserves 204/204 investment.

## 3. Evidence

A throwaway dd-markup serde (`dd/markup.py`, 380 LOC) was built via TDD on branch `v0.3-dd-markup-probe` and run against the full 204 Dank Experimental app_screen corpus.

### Dict-level round-trip parity

`parse_dd(serialize_ir(spec)) == spec` on every screen.

```
wins     : 204/204
serialize: 0 errors
parse    : 0 errors
mismatch : 0 dict diffs
time     : 7.5s for the full corpus
```

### Script-level parity (pixel parity by construction)

For each screen, `generate_figma_script(spec)` was compared between baseline IR and round-tripped IR (dict → markup → dict → script).

```
script-identical : 204/204
byte-equal       : 100%
time             : 15.6s for the full corpus
```

Byte-identical Figma scripts imply byte-identical walks and byte-identical `is_parity` verdicts. Pixel parity follows without needing to run the render sweep. The Figma bridge is deterministic given the same script; our proof is script-identity, which is strictly stronger than running the sweep once.

### Grammar extensions required to achieve this

Beyond vanilla KDL v2, three sentinel patterns were needed:

- `children <eid>*` — positional-arg list for element children (because KDL properties are single-valued; children are a list)
- `<name> { _entry ... _entry ... }` — list-of-dicts (for fills, strokes, effects, etc.)
- `<name> { _list_empty }` — empty-list marker to disambiguate from empty-dict `{}`

Underscore IR fields (`_node_id_map`, `_mode1_eligible`, `_corpus_source_node_id`, `_original_name`, `_composition`) round-trip as ordinary properties without special grammar — the prefix is preserved literally.

## 4. Cost comparison

### Aggressive cost

- Rewrite `dd/ir.py::generate_ir` to emit markup AST directly (not dict)
- Rewrite `dd/renderers/figma.py` (~1,800 LOC) to consume AST, not dict
- Rewrite `dd/compose.py::compose_screen` + providers (~29 funcs in dd/composition) for AST inputs
- Rewrite the 14+ renderer sites that read `_node_id_map`, `_mode1_eligible`, etc. from the dict
- Add AST-level testing equivalent to current dict-level coverage
- Risk: parity regression during refactor before we've landed any new user-visible feature
- Estimated effort: **multi-week refactor, with parity-recovery mid-way**

### Conservative cost

- Ship `dd/markup.py` as-is (already exists in throwaway form from this investigation — ~380 LOC to harden)
- Use it as a serde at LLM boundaries only
- **Zero renderer changes.** Pixel parity holds by construction (proven above).
- Drift pressure: any new property added to the dict IR must also get a serde path. New markup-only features (edit verbs, provenance) must also round-trip through dict IR if they affect render.
- Estimated effort: **days, not weeks**

### Aggressive upside foregone

- Single source of truth (no schema duplication)
- LLM edits could operate directly on the same structure the renderer walks (saves the serde hop)
- Every property automatically has a grammar representation (can't forget to add one)

These are real benefits, but they're **optimizations of maintenance**, not enablers of MVP features. We can achieve all v0.3 MVP goals (synthetic generation, edit grammar, verifier-as-agent) with conservative.

## 5. Decision

**Adopt conservative.** Evidence:

1. The proof gate is PASSED at the strongest level — script-identity, not just dict-identity, not just sampled pixels
2. Aggressive refactor cost ≫ conservative shipping cost
3. No MVP feature requires aggressive
4. Revisit is always available — the markup is already the lossless form; promoting it to canonical later is a contained refactor, not a rewrite

## 6. Consequences for downstream priorities

### Priority 1A (underscore fields)

**Resolved as bridge.** All five underscore fields round-trip as ordinary markup properties (verified). They don't need grammar-first-class treatment. Contract documentation still needed per field (ownership, semantics) but representation is decided.

### Priority 1B (grammar modes)

**Unchanged.** Dict-level proof doesn't speak to clustering. Still requires separate investigation.

### Priority 1C (RenderReport schema)

**Unchanged.** Unrelated to serde.

### Priority 3 (provider obsolescence audit)

**Scope shrunk.** With conservative, composition providers keep operating on dict IR. They don't need to learn markup. Obsolescence audit becomes: "which providers are obsoleted by v0.3's CRAG 3-mode cascade design?" — not "which providers need rewriting for a new IR?"

### Priority 4 (parity-gated branching)

**Unchanged.** Still mandatory — any serde change still needs 204/204 verification.

## 8. What could still invalidate this decision

If any of these emerge during MVP, revisit aggressive:

- A markup-only feature (e.g., a new provenance annotation type) whose properties must drive renderer behavior, forcing us to synchronize schema in two places indefinitely
- Performance hotspot in the dict-to-markup serde at scale (unlikely at 7.5s per 204 corpus)
- Edit operations that are painful to express against dict IR but natural against AST (ergonomics complaints should count as data)

None of these are predicted. If they emerge, the cost of promoting markup to canonical is still bounded (we already have the lossless serde).

## 9. Appendix — the grammar as it stands

Minimum dialect needed to round-trip today's IR:

```
// Top-level
version "1.0"
root "<eid>"
elements { element "<eid>" ... }
tokens { ... }
_node_id_map { map "<eid>" <int> ... }

// Element body
element "card-1" type="card" _mode1_eligible=true _original_name="Primary CTA" {
  children "text-1" "button-1"
  layout direction="vertical" gap=16 {
    sizing width=320.0 height="hug"
    position x=24.0 y=120.0
  }
  visual {
    fills {
      _entry type="solid" color="#F6F6F6"
    }
    cornerRadius=16
  }
}
```

Not identical to the aspirational `architecture-v0.3.md` §2 grammar (which adds `#`/`@` sigils, `->`/`&` references, named definitions). This is the **round-trip subset**. The aspirational grammar is a superset — we can grow into it feature by feature, with each extension going through the same round-trip proof.
