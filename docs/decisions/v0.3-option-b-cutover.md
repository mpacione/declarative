# v0.3 Decision: Option B Cutover (Markup-Native Renderer)

**Status:** Adopted 2026-04-18 (late-late session).
**Position:** dd markup IS the canonical IR. No dict-IR intermediate on the render path. Figma renderer (and future backends) walk L3 AST directly.
**Supersedes:** `docs/decisions/v0.3-canonical-ir.md` (both its original "conservative" position AND its 2026-04-18 "dict IR canonical on render path" revision).
**Revisit trigger:** if the markup-native renderer demonstrably loses semantic information the dict-IR projection preserved, AND the loss is not recoverable via per-node DB lookups. Not expected.

## 1. The question

Is dd markup the IR, or is it a projection of the dict IR with bidirectional bridges?

This question was asked during the investigation round (`v0.3-canonical-ir.md` §1). It was answered "dict IR is canonical on the render path; markup is L3 and lowers to dict IR" — Option A per `docs/spec-l0-l3-relationship.md` §3.2. Option B (L3-aware renderer) was deferred post-v0.3.

A 2026-04-18 late-session Figma sweep on the Option A path surfaced the architectural cost of the deferral: the markup round-trip requires **multiple compile-time side-channels** (`$ext.nid`, proposed `$ext.spec_key`) to preserve dict-IR identity across the lowering boundary. Those side channels are scaffolding for a system being demolished.

The question is re-raised: **adopt Option B now instead of accreting scaffolding.**

## 2. Evidence from the Option A attempt

### 2.1 Tier 1 and script-size-ratio passed

- `parse(emit(compress_to_l3(dict_ir))) == compress_to_l3(dict_ir)` — 204/204 on the corpus.
- Script-size ratio on 3 fixtures: 0.977 – 0.981 after extensive content-drift fixes (canonical-type classification, master-subtree inflation, parent-scoped EID lookup, `$ext.nid` channel).

### 2.2 Tier 2 byte-parity DID NOT hold

Byte-parity on the rendered Figma script — the actual Tier 2 claim in `spec-l0-l3-relationship.md:662-664` — was NOT achieved. Script-size ratios near 1.0 masked real divergence:

- Element-key counter drift between baseline `generate_ir` and the decompressor's AST walk. Same physical element keyed as `button-1` in baseline, `button-16` in round-trip.
- Content drift (`fill_mismatch`, `bounds_mismatch`, `effect_missing`) on ~25% of elements per screen.

### 2.3 Tier 3 pixel-parity sweep: 0/3 on smoke test

Running `render_batch/sweep.py --via-markup --limit 3` produced:

- Screen 118: 0.28 parity, 61 errors (missing_child, fill_mismatch, effect_missing).
- Screen 119: 0.28 parity, 61 errors.
- Screen 120: 0.41 parity, 63 errors.

Baseline sweep (no `--via-markup`) on the same screens: 3/3 `is_parity=True`. The markup round-trip is the regression; the baseline renderer is fine.

### 2.4 Root-cause analysis

Two drift categories:

1. **Dict-IR identity drift (48 `missing_child` errors on screen 181).** Baseline `generate_ir` assigns counter keys via a single DB walk; the decompressor's AST walk produces different counters because master-subtree inflation interleaves differently. Renderer propagates whichever counter key the spec had as Figma pluginData. Verifier fails to match nodes between walks.

   **Fix under Option A:** side-channel (`$ext.spec_key`) embedding the baseline counter into the markup. Pure scaffolding — no reader of markup outside the v0.3-specific lowering step cares.

2. **Content drift (fill/bounds/effect mismatches).** Real fidelity losses in the decompressor. `IR=#FFFFFF rendered=#000000` on a button fill, for example. Each error kind is a distinct decompressor bug.

   **Fix under Option A:** forensics pass on each drift class, patching the decompressor for every `_db_row_to_element` / `_expand_screen_subtree` / override-merge interaction.

### 2.5 Cost comparison (from `v0.3-canonical-ir.md` §4)

**Remaining Option A cost (to ship Tier 2/3 parity):**
- `$ext.spec_key` side-channel implementation + tests.
- Content-drift forensics (multiple drift classes).
- Figma sweep parity recovery iteration.
- Estimated: 1-2 weeks.

**Option B cost (full cutover):**
- Renderer rewrite — `dd/renderers/figma.py` is **2495 LOC** total (`generate_figma_script` is the 1011-line public entry; ~1500 LOC of helpers around it). ~30-40% of helpers are AST-neutral and reusable (JS escaping, font collection, prefetch, `__errors` channel). The ~1500 LOC of dispatch and property-emit code must be ported.
- `generate_ir` → `derive_markup` (~400 LOC; existing `compress_to_l3` is most of the work).
- Compose / providers / verifier rewrites (~500 LOC).
- Two-pass migration with parity-recovery window.
- **Estimated: 2–4 weeks.** The original 2–3 week estimate pre-dated the Option A sweep evidence showing even the preamble-level byte-parity is not trivially achievable and must be proven per-feature.

**Delta from Option A: still ~1 week additional work at the low end, up to ~2 weeks if preamble / Phase-1 / Phase-2 parity each surface non-obvious divergences. The architecture case stands — Option A carries scaffolding into Stages 2–6; Option B eliminates it.**

## 3. What Option B means concretely

Two stable representations:
- **L0+L1+L2 in the DB** — ground truth, lossless extract target, shared resources (component masters, token registry, asset registry, classifications, bindings).
- **L3 as markup AST** — in-flight IR for any single screen/component. Axis-polymorphic, reference-only values. Authoring, editing, synthesis, extraction output, render input — all one shape.

No dict IR anywhere on the render path.

Four primitive operations, one direction each:

```
Figma ──extract──▶ DB                (existing, unchanged)
DB ──derive──▶ markup AST            (replaces generate_ir)
markup AST ──render──▶ Figma JS      (replaces generate_figma_script)
markup AST ──persist──▶ DB           (synthesis + edit target)
```

One identity model: `Node.head.eid` (grammar §2.3.1). Written as Figma pluginData. Read by verifier. `-N` collision suffix for same-sanitized-name siblings. Auto-id `{type}-{n}` only when no legal IDENT derivable.

Values are references. The **renderer** is the boundary where references (tokens, component keys, asset hashes) resolve to backend-specific concrete values. Each backend (Figma, React, SwiftUI, Flutter) walks markup + resolves references against its flavor of the catalog. Same markup, different resolution.

## 4. What changes in the code

See `docs/DEPRECATION.md` for the full list. Summary:

**Rewritten / renamed:**
- `generate_ir(conn, sid) → dict` becomes `derive_markup(conn, sid) → L3Document`.
- `generate_figma_script(spec, ...)` becomes `render_figma(doc: L3Document, conn, ...)`.
- `dd/compose.py` + `dd/composition/*` consume AST instead of dict.
- `dd/compress_l3.py` core logic becomes `derive_markup` — same per-axis derivation, different output shape.

**Deleted at cutover (M6):**
- `dd/decompress_l3.py`.
- `tests/test_decompress_l3.py`, `tests/test_markup_render_pipeline.py` (Option A tests).
- `$ext.nid` emission.
- `--via-markup` flag (becomes default).

**Preserved:**
- `dd/markup_l3.py` (parser, emitter, AST, semantic passes).
- Grammar spec, L0↔L3 spec (§3 revised).
- `dd/archetype_library/*.dd`, archetype tests.
- Fixture files, golden snapshots.
- Main sweep harness (reconfigured to the single remaining path).

### 5.1 Why M1 has four sub-milestones

A pre-M1 research round (2026-04-18, evening) stress-tested the plan against the actual state of the repo. Three findings reshaped the M1 gate:

1. **Option A never achieved byte-parity even at the preamble level** (`diff scripts-markup/181.js scripts/181.js` shows a dropped `Inter Variable Regular` font load). "Byte-parity on screen 181" as a single-step gate was tighter than Option A's historical best (ratio 0.95–1.05 at Stage 1.5c; 0/3 at Tier 3 pixel parity). Softening M1's gate to pipeline-health and tightening M2 to full byte-identity gives a realistic ladder.

2. **The 48 `missing_child` verifier errors on screen 181 were eid drift, not node-creation failure.** `dd/verify_figma.py` matches by `eid_map[eid]` populated from the renderer's `M[eid] = n0.id` assignments — not via Figma `pluginData`. Option A's baseline `generate_ir` and the decompressor's AST walk produced different element-key counters (`button-5` vs `button-16`), causing verifier misses even though nodes were created correctly. In Option B, the compressor and renderer walk the same AST deterministically, so eids agree by construction. **This entire failure class disappears in Option B** — it does not need fixing or working around.

3. **The eid↔nid identity bridge is metadata, not grammar.** `spec["_node_id_map"]` exists in Option A as a dict-level side-car. Option B replaces it with a function-return side-car: `derive_markup(conn, sid) → (doc, nid_map)`. This is the ONE thing `$ext.nid` was trying to do in-grammar; keeping it out of the grammar is the full fix, not a partial one.

These findings are load-bearing for M1a's design: without the explicit side-car, M1b's preamble-byte-parity gate cannot be reached (font collection traverses per-node DB reads keyed on nid).

## 6. Parity-protection during migration

The Option A machinery stays operational in CI throughout M1–M5. Both paths run; an A/B harness asserts byte-identical Figma script output on every screen. The Option A path IS the reference against which Option B's correctness is measured.

At M6, a single atomic cutover PR:
- Switches all production call sites to `render_figma(doc, conn)`.
- Deletes the Option A code per `DEPRECATION.md`.
- Updates tests.

If M6 lands clean (204/204 pixel parity), the v0.3 ship criterion is met.

## 7. Why this is the right call now (not after v0.3 ships)

**Scaffolding grows by pull.** Every Stage 2+ feature built on Option A (pattern expansion, synthetic tokens, synthesis) would take a dependency on the dict-IR intermediate. Rewriting under Option B gets harder, not easier.

**The 1-week delta is front-loaded.** Option A needs `$ext.spec_key` + content-drift forensics; Option B needs the renderer rewrite. Both are work that doesn't currently exist. Doing B means that work produces the final architecture, not transitional scaffolding.

**Interview / external eyes.** The tagged checkpoint `pre-markup-baseline` preserves the current clean dict-IR state on `main` for sharing. Current `v0.3-integration` is tagged `option-a-complete` as archaeology. The Option B work proceeds on `v0.3-integration` from that tag forward. No branching confusion; tags hold the rollback points.

**MLIR analogue.** Tier 0 §1 frames the architecture as LLVM MLIR applied to design artifacts. MLIR's power comes from a single core IR that all passes and backends consume. Option A violates that — dict IR is a second representation that the renderer happens to prefer. Option B restores the single-core-IR invariant.

## 8. What this does NOT change

- 204/204 round-trip pixel parity remains non-negotiable at every commit (Tier 0 §3.1 / v0.3 Invariant #1). The Option A path guarantees this during the migration window.
- Tier 0 §3.3 "no raw values in the IR" is strengthened, not weakened — value resolution happens at one place (the renderer) with per-backend rules.
- ADR-001 capability-gated emission — now THE mechanism for both emission and constrained decoding, across all backends.
- `docs/spec-dd-markup-grammar.md` — the grammar is already Option B-ready. No changes required.
- The archetype migration work (Stage 2 archetypes, 12 `.dd` files) — already markup-native.
