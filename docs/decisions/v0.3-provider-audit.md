# v0.3 Decision: Provider Obsolescence Audit (`dd/composition/*`)

**Status:** Proposed 2026-04-18.
**Dependent on:** `docs/decisions/v0.3-canonical-ir.md` (Priority 0 conservative).
**Related:** Priority 3 per `docs/continuation-v0.3-next-session.md` §4.

## 1. Scope shrunk post-Priority 0

With Priority 0 resolved conservative (dd markup = serde over dict IR; dict IR stays canonical), providers keep operating on dict IR. No IR-format rewrite needed anywhere. The audit narrows to: **what does the v0.3 CRAG 3-mode cascade (SCREEN_EDIT / COMPOSITION / SYNTHESIS, `architecture-v0.3.md` §5) obsolete or rewrite in the existing code?**

Tag meanings:
- **SURVIVES** — load-bearing as-is, no v0.3 changes required
- **REWRITES** — concept survives, implementation changes for CRAG/v0.3
- **OBSOLETED** — v0.3 design absorbs or eliminates; deletion candidate
- **UNCLEAR** — needs more investigation

## 3. Quantitative summary

```
SURVIVES   : 11 top-level + 3 providers  = 14 files unchanged
REWRITES   : 2 top-level + 1 provider   + compose.py sections   = 4 modules touched
OBSOLETED  : 2 top-level + deletion candidates inside compose   = ~335 LOC deletable
UNCLEAR    : 0
```

Roughly two-thirds of `dd/composition/*` ships unchanged. Remaining third is rewritten, not rebuilt — the semantics of CRAG's three modes map onto existing code concepts (plan-then-fill, corpus retrieval, splice vs re-emit, provider priority).

## 4. MVP sequencing implications

The CRAG 3-mode cascade work is **NOT blocked on provider rewrites**. Day 1 of MVP can proceed with:

- Priority 0 complete (markup as serde — done)
- Priority 1A contract docs (documentation only)
- The REWRITES set gets addressed in Stage 2–3 of the MVP, feature-by-feature, each landing with its own Tier 2 parity check

Explicit ordering risk: `compose.py::compose_screen` rewrite is a high-touch refactor. If Tier 3 parity sweep is slow (~20 min per iteration), we want compose_screen changes batched, not drip-fed. Suggested sequencing:

1. **Stage A — CRAG scaffolding.** Introduce `dd/composition/crag_cascade.py` with SCREEN_EDIT / COMPOSITION / SYNTHESIS stubs that delegate to existing code. No behavior change. Land with 204/204 green.
2. **Stage B — SCREEN_EDIT path.** Implement on top of existing `corpus_retrieval` + `project_ckr`. Lowest risk of parity regression.
3. **Stage C — SYNTHESIS path.** Route through new archetype + universal fallback. Parallel to existing Mode-3-with-corpus path for A/B comparison.
4. **Stage D — COMPOSITION path.** Hardest — replaces splice with re-emission. This is where the markup grammar actually earns its keep; `parse_dd`/`serialize_ir` get used for provenance boundary verification.
5. **Stage E — deprecation.** Once the three modes ship green, delete the matrix_* modules and the splice logic in compose.py.

## 5. What this doesn't cover

- `dd/renderers/figma.py` is NOT in scope here — that 1,800-LOC file is SURVIVES per Priority 0 (renderer walks dict IR unchanged). Audit noted it as load-bearing; no v0.3 changes required.
- `dd/classify.py` and the `classify_*` modules sit upstream of composition (they populate the DB); they're independent of CRAG redesign.
- Extract-side code (`dd/extract_*`, `dd/ir.py`) likewise independent.
