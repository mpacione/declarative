## 1. Problem

The v0.3 verifier-as-agent design (`architecture-v0.3.md` §7) expects the verifier to return **both** structural observations (current `RenderReport`) **and** prescriptive edits (new — expressed in the dd-markup 7-verb edit grammar).

Reviewer 3 flagged that `dd.boundary.RenderReport` is a `frozen=True` dataclass and adding `proposed_edits` is a schema change affecting all consumers. The consumers are few but real:

- `dd/verify_figma.py` — constructs `RenderReport`
- `dd/cli.py` — prints and emits JSON (`dd verify --json`)
- `render_batch/sweep.py` — parses `dd verify --json` output
- `tests/test_boundary_contract.py` + others — structural tests

## 2. Three positions considered

### A. Extend RenderReport with a nullable field

```python
@dataclass(frozen=True)
class RenderReport:
    backend: str
    ir_node_count: int
    rendered_node_count: int
    errors: list[StructuredError]
    proposed_edits: list[MarkupEdit] | None = None  # new
```

- Semantics: `None` = verifier does not propose edits (current Figma verifier, CI use). `[]` = verifier ran and found nothing to change. `[edit, ...]` = verifier-as-agent prescription.
- Backward-compatible for construction (default = None).
- All current consumers continue working untouched.
- Frozen-dataclass-with-default is the standard Python additive pattern.

### B. Wrapper class

```python
@dataclass(frozen=True)
class VerifierOutput:
    report: RenderReport
    proposed_edits: list[MarkupEdit] = field(default_factory=list)
```

- Semantics: `VerifierOutput` is the "observation + prescription" bundle; `RenderReport` stays the observation-only struct.
- Cleaner conceptual separation.
- Every consumer of verifier output needs the wrapper; adds one level of access.
- Migration: verify_figma returns `VerifierOutput(RenderReport(...), [])` for parity-only checks.

### C. Compatibility shim

Keep `RenderReport` frozen and untouched. Introduce `RenderReportV2` with `proposed_edits`. Provide a coercion `to_v1(v2: RenderReportV2) -> RenderReport`. Consumers opt in when ready.

- Most conservative.
- Temporary zoo of two types with identical semantics 99% of the time.
- Leads to duplicated code in verify_figma / cli.

## 3. Decision

**Option A: extend RenderReport with `proposed_edits: list[MarkupEdit] | None = None`.**

Rationale:

1. The concept IS part of the verifier contract. v0.3 treats verification and repair as one loop (ADR-007's unified verification channel), so bundling observation + prescription in one struct matches the architectural intent.
2. All current consumers work untouched because of the default.
3. The `None` vs `[]` distinction captures "verifier doesn't do this" vs "verifier does this but had nothing to propose" — useful for signal separation in sweep summaries.
4. If later the wrapper pattern (Option B) turns out to be cleaner, a refactor is contained: it's one struct's shape change.

**What to add:**

```python
# dd/boundary.py — new
@dataclass(frozen=True)
class MarkupEdit:
    """A single edit from the dd-markup 7-verb grammar.

    Verb ∈ {"set", "append", "insert", "delete", "move", "swap", "replace"}.
    Arguments are verb-specific; we keep them loosely typed here because
    the canonical form is the dd-markup source text in `source`.

    `source` is the exact markup fragment the verifier emitted (parseable
    via `dd.markup.parse_dd`). The parsed form is redundant with source
    but pre-computed for consumers that want structured access.
    """
    verb: str
    target_eid: str
    source: str  # canonical markup
    parsed: dict[str, Any] | None = None  # pre-parsed payload, optional

    # Optional severity / confidence — verifiers that know will populate,
    # others leave as None. Used for edit-ranking in repair loops.
    severity: str | None = None  # "error", "warning", "suggestion"
    confidence: float | None = None  # 0.0–1.0
```

Plus:

```python
# dd/boundary.py — RenderReport extension
@dataclass(frozen=True)
class RenderReport:
    backend: str
    ir_node_count: int
    rendered_node_count: int
    errors: list[StructuredError]
    proposed_edits: list[MarkupEdit] | None = None  # NEW
    # ... is_parity and parity_ratio unchanged
```

## 4. Migration plan

1. **Land the schema change.** Add `MarkupEdit`; add `proposed_edits` to `RenderReport` with `None` default. Zero consumer changes required.
2. **CLI JSON output.** Update `dd verify --json` to include `proposed_edits` when non-None. Downstream parsers in `render_batch/sweep.py` already JSON-parse and ignore unknown fields (verified — `process_screen` accesses `report["is_parity"]`, `report["parity_ratio"]`, `report["ir_node_count"]`, `report["rendered_node_count"]`, `report["errors"]`).
3. **verify_figma.py.** Returns `RenderReport(..., proposed_edits=None)` — explicit None, documents "this verifier is observe-only, not an agent."
4. **Verifier-as-agent impl.** New module `dd/verify_agent.py` (or similar) implements `RenderVerifier` protocol, returns `proposed_edits=[MarkupEdit(...)]`. Uses dd-markup edit grammar under the hood.
5. **Test the grammar round-trip.** Each `MarkupEdit.source` string must be parseable by `dd.markup.parse_dd` (when the parser supports edit verbs — currently parses structural markup only; see §6 follow-ups).

## 5. What this does not break

- `is_parity` / `parity_ratio` logic unchanged — `proposed_edits` doesn't factor into parity
- JSON shape is additive (consumers that don't know about `proposed_edits` ignore it)
- `tests/test_boundary_contract.py` tests for `errors` / `is_parity` invariants don't touch the new field
- `render_batch/sweep.py` summary is unchanged in shape

## 6. Follow-ups

- [ ] Add `MarkupEdit` type to `dd/boundary.py`
- [ ] Extend `RenderReport` with `proposed_edits` field (default None)
- [ ] Update `dd verify --json` to serialize proposed_edits when populated
- [ ] Extend `dd/markup.py` parser to handle edit-verb nodes (currently parses structural IR only). Grammar: `set @eid prop=value`, `append to=@eid { ... }`, etc.
- [ ] Write `tests/test_markup_edit_grammar.py` — progressive TDD test like the round-trip probe, for each of the 7 verbs
- [ ] New verifier-as-agent module (post-MVP-Stage-2)

## 7. What could still invalidate this decision

- If `proposed_edits` turns out to be heavy (hundreds of edits per screen per verifier run), the frozen dataclass carrying them becomes awkward for diffing/hashing. Then revisit Option B (wrapper).
- If multiple verifiers want to emit differently-structured suggestions (edits for Figma, patches for React, etc.), the uniform `MarkupEdit` type may become a bottleneck. Revisit with a generic `Prescription` protocol.

Neither is predicted. Ship Option A.
