## Part 1 — The architectural decisions this research produced

These are the decisions that survived all 18 threads. Each is load-bearing; each is citable below.

1. **The IR / markup is axis-polymorphic, not level-structured.** Every node lives in a 5-axis specification space (Structure / Content / Spatial / Visual / System). Any subset of axes can be filled. Density is a property of each node, not a stage of the pipeline.

2. **The IR never holds raw values.** Every value is a token reference. Real tokens come from the user's design system; **synthetic tokens are clustered from raw values during extraction** (ΔE grouping on colors, histogram on dimensions) and preloaded from universal defaults (shadcn-style) for cold-start synthesis. LLMs never hallucinate values — they choose from a known vocabulary.

3. **One grammar serves all artifacts.** The same markup grammar expresses IR state, edits, patterns, themes, archetypes, constraints, and verifier feedback. Generator, verifier, user, and renderer all speak the same language. Single constrained-decoding grammar.

4. **Named definitions with layered overrides are first-class.** `define` + reference + axis-specific override at call site. Patterns, themes, archetypes, component libraries, edit templates all collapse into one mechanism.

5. **The Pattern Language is a pipeline optimization, not a markup constraint.** L0→L3 descent is the cheapest way to explore during synthesis (branch at L1, prune, elaborate, style). But the markup itself doesn't require the order — a designer can author at any density and iterate at that density.

6. **Synthesis uses a CRAG-style three-mode cascade.** SCREEN_EDIT (strong donor match — edit the donor IR directly) → COMPOSITION (partial matches — archetype skeleton + role-indexed subtree exemplars, never stitch raw) → SYNTHESIS (no match — skeleton + universal catalog). All three converge to the same edit grammar; they differ only in what starting IR is handed to it.

7. **Verifier-as-agent uses the shared edit grammar.** Gate 1 (structural parity) + Gate 2 (density check) + Gate 3 (Jaccard, rank-only) + Gate 4 (pairwise VLM, rank-only). Best-of-N variant generation beats sequential refinement. Critic emits edit operations, not scalar scores. Fresh-context critic prevents reward hacking.

8. **Multi-target extension is a catalog schema change, not an architectural rewrite.** Per-target metadata (Code Connect pattern) on catalog entries + per-target token resolvers (Style Dictionary pattern). IR + markup + renderer core remain target-agnostic.

9. **Extract emits ground truth, patterns are detected by a separate optimization pass.** Round-trip invariant requires extractor to preserve what's in the source (INSTANCE → `->`, FRAME → inline). Auto-pattern-detection runs post-extract with Rule of Three (N ≥ 3) and suggests — never applies — definitions. Users explicitly promote inline → pattern → external component.

10. **Scoping is one-namespace-per-file + mandatory-alias imports + hard-error on cycles.** Elm + W3C Design Tokens precedent. Three-color DFS for cycle detection. No re-exports, no version pinning in syntax, no partial-resolution-of-missing-refs.

## Part 2 — The markup-design tradeoff research (4 threads)

These four agents were dispatched together on 2026-04-18 to resolve the five concrete tradeoffs raised by the user. Each returned ≤700 words with a comparison table and a concrete recommendation.

### Thread 1 — Scoping, imports, and circular-reference detection

**Question:** How do markup/configuration DSLs with reusable definitions handle scoping, imports, and cycle detection?

**Systems surveyed:** ES Modules, Python, Rust, Sass (@use/@forward/@import), Jinja2, Elm, KDL, W3C Design Tokens, YAML anchors, Bazel Starlark.

**Dominant patterns:**
- **Explicit-import + per-file namespace** (ES/Rust/Sass `@use`/Starlark). Every reference is traceable to one `use` line. Safe for tooling; verbose for LLMs.
- **Alias-by-reference, no import verb** (YAML anchors, W3C design tokens, JSON Pointer). One namespace, references resolve by name. Minimal syntax, great LLM emission, requires strict cycle check.
- **Template inheritance** (Jinja `extends`, Sass partials). One "parent" slot plus `include`/`import` for composition. Matches "base + overrides."

**Decision:** Hybrid of patterns 2 + 3. One namespace per file declared at top; `use "path" as alias` with alias mandatory; dot-paths for references (`pattern.product-card`); `@` prefix for references; hard-error on cycles via three-color DFS (not Tarjan — we only need cycle existence plus path for the error message); last-file-definition wins with warning; no re-exports; skip version pinning in syntax.

**Risk flagged:** Path hallucination is the biggest LLM failure mode. Mitigations: (a) inject full list of available libraries into the system prompt every call; (b) constrained decoding grammar restricts `use` paths to the known set.

### Thread 2 — Hierarchical identifier generation

**Question:** When a reusable definition is instantiated multiple times, how do descendant nodes get stable unique IDs for later addressing?

**Systems surveyed:** React keys/Fiber, Figma instance nodes, SVG `<use>` and Shadow DOM, Web Components CSS Parts, Sass mixins, hygienic Lisp macros, Handlebars partials, MLIR `SymbolRefAttr`, SwiftUI structural identity.

**Dominant patterns:**
- **Scoped-symbol-table composition** (MLIR, Racket, Handlebars `../`). Each expansion opens a named scope; addressing is `outer::inner::leaf`. Qualified names always valid; short names resolve via scope lookup.
- **Explicit opt-in parts with rename-on-export** (CSS Shadow Parts, Lisp `export`). Authors mark which descendants are addressable.
- **Structural identity with manual override escape hatch** (React `key`, SwiftUI `.id()`, Figma instance prefix). System auto-generates positional id; author overrides at call site when structure breaks.

**Decision:** Combine all three. Auto-generated `scope@N` suffix by sibling index as default; explicit `as name` or `#id` at call site for stable aliases; propagating prefix (`pattern.product-card name=hero`) for bulk naming; `/` path separator; wildcard selectors (`grid/*/buy`); constrained decoding over live symbol table to eliminate LLM typos.

**Algorithm (from the research):**
```
expand(definition D, call_site C, parent_scope P):
  if C has explicit alias "as NAME":  scope_name = NAME
  elif C has "#NAME":                  scope_name = NAME
  else:
      base = D.name
      n    = count_siblings_with_same_def(P, D) - 1
      scope_name = f"{base}@{n}"
  scope = P.open_child(scope_name)
  for node in D.body:
      local_name = node.id or default_from_type(node)
      scope.bind(local_name, expand(node, ...))
  return scope
```

**Risk flagged:** Reorder-induced drift on `@N` suffixes. Mitigation: recommend explicit `as NAME` whenever an address will be reused; lint on `@N`-suffixed edit targets.

### Thread 3 — Parametrization (slots vs args vs overrides)

**Question:** Slots, named arguments, or some other mechanism for varying reusable templates? Is there a uniform mechanism?

**Systems surveyed:** React props + children, Web Components `<slot>`, Vue scoped slots, Svelte 5 snippets, SASS `@mixin`, Jinja2 macros, Handlebars partials, Figma Component Properties + Variants.

**Dominant patterns:**
- **Args-only** (Handlebars partials). Everything is a named scalar. Easiest for LLMs, lowest ceiling. Fails on structural change.
- **Slots-only** (early Web Components). Everything is a subtree. Forces "change the title" into `<slot name="title">Hello</slot>` — verbose, easy to mis-target.
- **Hybrid** (Vue, React+children, Svelte 5, Figma). **Every mature production system converges here.** Scalar-vs-tree is a genuine ontological distinction, not a skin on the same thing.

**Decision:** Three primitives, non-overlapping responsibility:
- **Typed scalar args** (`title: text`, `accent: color`) — for Content and single-value Visual/Spatial axes
- **Named slots with optional defaults** (`slot action = button/primary(...)`) — for Structure axis variation
- **Path-addressed property overrides** (`pattern.product-card card.fill=black`) — for partial axis overrides that don't warrant a declared arg

Override rules: call-site > definition default (fail-open, no error); slots and scalars replace by value; path overrides merge at property level; maps shallow-merge, arrays replace. This matches Figma's Component Property model exactly and maps cleanly onto the existing override-decomposition machinery.

**"Everything is slots" thought experiment:** Dies ergonomically. Scalars give axis scoping, better LLM constrained decoding, and path stability across Structure-axis refactors. Slot-only model loses all three.

**Sources:**
- React render-props: https://react.dev/reference/react/cloneElement
- Vue scoped slots: https://vuejs.org/guide/components/slots.html#scoped-slots
- Svelte 5 snippets: https://svelte.dev/docs/svelte/snippet
- SASS mixins: https://sass-lang.com/documentation/at-rules/mixin/
- Figma Component Properties: https://help.figma.com/hc/en-us/articles/5579474826519

### Thread 4 — Extract-vs-author distinction

**Question:** How do systems distinguish "instance of externally-defined component" from "locally-authored reusable template"? Different round-trip semantics required.

**Systems surveyed:** Figma InstanceNode/ComponentNode, Sketch library symbols vs local, Penpot components vs groups, Adobe XD components vs repeat grids, Framer code vs canvas components, Storybook CSF3, React components vs SCSS @mixin.

**Dominant patterns:**
- **Dual-node-type model** (Figma, Sketch, Penpot, XD). Two distinct types — reference-with-overrides (INSTANCE) vs structural (FRAME/GROUP). Detach is type transition INSTANCE → FRAME. Promotion is FRAME → COMPONENT.
- **Key-indirection for external references** (Figma, Sketch library symbols). Global `key`, not local id.
- **Snapshot vs live-link** (SCSS vs React). The most important semantic decision and should be explicit in grammar.

**Decision:** Two distinct sigils:
- `-> button/primary/lg` — external component instance (component_key/library_ref). Source of truth: upstream component. Round-trip preserves the reference.
- `& product-card` — local pattern reference (in-document `define`). Source of truth: in-document definition. Expansion is inline.
- (no sigil) — inline structural subtree.

Disambiguation is deterministic by registry lookup: `->` requires a registry entry; `&` requires a local `define`. Extractor emits `->` for Figma INSTANCE nodes with `component_key`, emits inline for FRAME nodes — **never auto-collapses duplicates at extract time** (preserves round-trip invariance).

Pattern detection is a **separate optimization pass** that SUGGESTS (never applies):
```
1. Canonicalize each IR subtree: strip eids, positions, DTCG-token values.
2. Hash canonical form (structural hash, à la jsinspect).
3. Group by hash. For groups with N >= 3 (Fowler's Rule of Three), suggest.
4. Divergent leaf-values become the pattern's parameter list.
```

**Risk flagged:** Detach cascade on nested instances (Figma has this edge case). Solution: mirror Figma's explicit `detachInstance()` semantics.

**Sources:**

## Part 3 — The earlier architecture research threads (10 earlier)

Dispatched earlier in the session; summarized here for continuity. Each agent returned a full report with sources.

### Retrieval + cascade

**Finding:** The three-mode cascade we proposed (SCREEN_EDIT / COMPOSITION / SYNTHESIS) is **Corrective-RAG (CRAG) applied to structured design.** Published, cited, has known properties. Multi-aspect retrieval (archetype + structural + semantic) beats single-vector at sub-1K corpus scale. Raw subtree stitching produces "Frankenstein" output; **always re-emit through the LLM with donors as in-context exemplars.**

**Key papers:** CRAG (arXiv 2401.15884), Self-RAG (Asai 2024), PrototypeAgent (arXiv 2412.20071), LayoutPrompter (NeurIPS 2023), UISearch (arXiv 2511.19380).

### Edit grammars

**Finding:** Typed DSL + constrained decoding wins for domain-specific structured edits. JSON Patch hits known failure modes (array-index arithmetic, related-update blindness). **Keep a 4–5 primitive grammar** (Anka 2025: 99.9% parse / 95.8% task on Haiku with zero prior training). Reference by stable eid, never positional index. Imperative verbs + keyword args (not `::` punctuation).

**Key sources:** JSON Whisperer (arXiv 2510.04717), JSONSchemaBench (arXiv 2501.10868), Anka DSL (arXiv 2512.23214), Aider edit formats (https://aider.chat/docs/unified-diffs.html), XGrammar (arXiv 2411.15100).

### Multi-target catalog

**Finding:** Use DTCG + Style Dictionary pattern for tokens (platforms live in config, not in tokens). Use Code Connect pattern for components (one-to-many via filesystem, per-target file). **Per-target JSON sub-objects on catalog rows**. Don't build your own Mitosis; lean on Mitosis for framework-codegen if we ever need it.

**Key sources:** DTCG 2025.10 stable spec (https://www.designtokens.org/tr/drafts/format/), Style Dictionary (https://styledictionary.com), Figma Code Connect (https://developers.figma.com/docs/code-connect/), Mitosis (https://mitosis.builder.io/), Diez post-mortem (https://thoughtbot.com/blog/saying-no-to-diez-for-now).

### Verify + refine loop

**Finding:** **Pearson r = −0.96 quality-ceiling effect** (GameUIAgent): refinement plateaus below a threshold — regenerate instead. **Never feed VLM score as loss to same model** (Pan 2024 shared-context reward hacking). Use pairwise >> pointwise (arXiv 2504.14716). ReLook's "forced optimization" (monotone-only accept) is the only architecture that provably avoids divergence. Tiered ladder, not scalar score.

**Key sources:** GameUIAgent (arXiv 2603.14724), ReLook (arXiv 2510.11498), Spontaneous Reward Hacking (arXiv 2407.04549), LLMs Cannot Self-Correct Reasoning Yet (ICLR 2024 arXiv 2310.01798), AB-MCTS (arXiv 2503.04412), UICrit (UIST 2024 arXiv 2407.08850).

### Intent parsing

**Finding:** SpecifyUI-like structured intent schema outperforms unstructured prompts. Hybrid retrieval key: hard filter (archetype/platform/domain) as SQL predicate + soft similarity (mood/style/density) as embedding. Minimum intent schema: 6 fields + confidence per field. Clarifying questions: budget 1 per prompt max, EVPI-gated. Silent defaults are what Stitch/v0/First Draft do; research systems ask.

**Key sources:** SpecifyUI (arXiv 2509.07334), ASK (ACL 2025 Industry), QuestBench (arXiv 2503.22674), EVPI / SAGE-Agent (arXiv 2511.08798).

### Multi-donor composition

**Finding:** Never raw-stitch. Always re-emit through LLM with donors as structured sketches (not literal subtrees). Coordinator + serialized sub-agents + shared cache (PrototypeAgent pattern). Coherence enforced in layers: retrieval-time (strip absolutes), pre-prompt (theme unification, sizing normalization), re-emission (LLM), post-audit (deterministic eid uniqueness + token closure + FILL/HUG cascade), post-render (VLM).

**Key sources:** PrototypeAgent (arXiv 2412.20071), DesignCoder (arXiv 2506.13663), LayoutPrompter (NeurIPS 2023), CAL-RAG (arXiv 2506.21934), ARCS (arXiv 2504.20434).

### Evaluation methodology

**Finding:** Static fidelity scoring is insufficient. Pairwise VLM beats pointwise (MT-Bench position-bias literature). Paired bootstrap for significance (arXiv 2511.19794). 60-pair test set minimum for 0.8 power at Δ ≥ 0.4. When static says "identical" but VLM says "better", escalate via (a) pairwise re-judgement at position swap, (b) 3× VLM re-run, (c) trust VLM. UICrit's 55% gain from 8-shot joint-task-and-visual-similarity sampling.

**Key sources:** UICrit (UIST 2024), MLLM as UI Judge (arXiv 2510.08783), Design2Code (arXiv 2403.03163), Trust or Escalate (ICLR 2025), Judging the Judges (arXiv 2406.07791), Paired Bootstrap Protocol (arXiv 2511.19794).

### Pattern mining from corpus

**Finding:** At 204-screen scale, **skip embeddings — use skeleton-string clustering + frequent-subtree mining + LLM-labeling**. Our DB already has 338 screens → 73 distinct skeleton strings (Zipfian, top 8 cover 50%). Don't train Screen2Vec or Graph4GUI (both need 15K+ screens). Mine slots first, then archetypes, then style params (inverts Enrico/RICO order because slots are the generation grammar).

**Key sources:** Screen2Vec (arXiv 2101.11103), Graph4GUI (arXiv 2404.13521), Rico (CHI 2017), Enrico (MobileHCI 2020), TnT-LLM (arXiv 2403.12173), TELEClass (arXiv 2403.00165), pq-gram tree-distance (ECAI 2020).

### Constraints as first-class

**Finding:** Constraints live in three enforcement layers, not one: grammar at emission (decoding) + library vocabulary at naming + audit at render. Brand colors / WCAG contrast / platform / grid = minimum viable set. Mine corpus to derive defaults (delta-E clustering for palette, histogram for grid/type scale, 5th-percentile contrast as observed floor). LayoutFormer++ decoding-space restriction is canonical for hard-layout rules.

**Key sources:** DTCG 2025.10, LayoutFormer++ (arXiv 2208.08037), v0 March-2025 system prompt (WCAG defaults, shadcn, lucide-react), llguidance (https://github.com/guidance-ai/llguidance), Correctness-Guaranteed Code Generation (arXiv 2508.15866), Stark + axe for Designers.

### Verifier-as-agent + provenance-aware markup

**Two connected findings.** (a) Anthropic's own Artifacts explicitly defer verification to the user. **GUI-Critic-R1** is the closest prior art for "verifier emits the same action vocabulary as the generator" — no one has yet published the full "same grammar" version. (b) For provenance-aware markup, KDL v2 + DTCG `$extensions` + Pkl `amends` is the right substrate. Node-level provenance inherits down subtrees; per-value provenance via `#[...]` trailer only when interesting.

**Key sources:** Claude Computer Use (https://platform.claude.com/docs/en/agents-and-tools/tool-use/computer-use-tool), Anthropic web-artifacts-builder skill (https://github.com/anthropics/skills/blob/main/skills/web-artifacts-builder/SKILL.md), UI2Code^N (arXiv 2511.08195), METAL (ACL 2025), GUI-Critic-R1 (arXiv 2506.04614), OS-Oracle (arXiv 2512.16295), KDL v2 (https://kdl.dev/spec-v1/), Pkl (https://pkl-lang.org/), Hazel Grove (POPL 2025).

### Thread 13 — Enterprise internal tooling

**Killed attempts:**
- **Airbnb Lona** (2019, archived) — required designers to leave Sketch/Figma → zero adoption
- **Google Relay** (sunset April 30 2025) — pre-LLM, rigid, tightly coupled to Figma hierarchy

**Current enterprise choices:**
- Uber uSpec (2026) — deliberately chose LLM-as-translator, NOT an IR
- Shopify/Atlassian/Spotify/Microsoft/Adobe — tokens + hand-maintained per-platform libraries, no compiler

- Stripe Sail team dissolved March 2025

**Research:**
- Apple SQUIRE (UIST 2025) — research-only formal IR, closest academic match

**Signal:** Industry consensus is "LLM-as-translator + tokens-as-source-of-truth" over formal IRs. Pre-LLM IRs failed (Lona, Relay). Post-LLM IRs with verification channels are the contrarian bet. Our architecture directly addresses the specific failure modes that killed Lona/Relay.

### Thread 14 — Academic research frontier (2024–2026)

**Closest published matches:**
- **Apple's Athena** (arXiv 2508.20263, Aug 2025) — "storyboard + data model + GUI skeletons" maps onto our L0→L3. No Figma ingest, no round-trip.
- **Zhejiang's SpecifyUI/SPEC** (arXiv 2509.07334) — two-level IR + RAG over 2K pairs. Web-only, screenshot input.

**Frontier consensus (2025):** "Scaffolded IRs" as the accepted frame. Apple/SQUIRE, Zhejiang/SPEC, UICopilot's hierarchical decomposition, UI Grammar + LLM — all converge on the same bet that LLMs can't one-shot UIs and require a typed intermediate layer.

**We're parallel-to-slightly-ahead on architecture**, meaningfully ahead on:
- Round-trip fidelity as first-class metric (no group reports this)
- Figma as typed source of ground truth (everyone else screenshots)
- Corpus scale coupled to typed IR (SpecifyUI 2K vs our 42,938 classified)
- Multi-target production renderer (academia targets React/HTML only)

**Researchers to watch:** Alan Leung / Jason Wu / Ruijia Cheng / Titus Barik (Apple); Liuqing Chen / Yunnong Chen (Zhejiang); Yuwen Lu (Notre Dame → Apple intern); Yue Jiang (Aalto).

### Thread 15 — Design tool roadmaps

**Figma explicitly chose NOT to build a typed IR.** Code Layers uses React-as-runtime. Code Connect is key-value snippet lookup. MCP is filtered context, not grammar. No round-trip fidelity metric. Direction: deeper React fidelity + more Code Connect languages. Not multi-target generative compilation.

**Tool coverage for (bidirectional + multi-target + typed IR):**
- No tool ships the full triple
- Closest to any two: Figma (bi-dir + AI, single-target), Builder.io (multi-target + AI, forward-only web), Penpot (typed tokens + open standard, no compilation yet)

**Dark horse:** Penpot — SVG-native, W3C DTCG-native (first to fully adopt Oct 2025), OSS, MCP on 2026 roadmap. Under-resourced but architecturally positioned. Runner-up: Builder.io + Mitosis.

**Our 12–18 month window:** Native (SwiftUI/Compose) from a single design + structural round-trip fidelity — both unclaimed.

### Consolidated competitive verdict

**Honest:** The specific combination is genuinely unoccupied at shipping scale. Two failed enterprise attempts (Lona, Relay) set the bar for what NOT to do; our architecture addresses both failure modes. Academic consensus has converged in our direction in 2025 — we're aligned, not isolated. **Our differentiators are the 42K-classified corpus, round-trip-verified parity, multi-target including native, and verification-channel-as-grammar** — these are load-bearing and hard to replicate quickly.

**Timeline:**
- 6 months: Builder.io could add bidirectional + SwiftUI to Mitosis
- 12 months: Figma Code Connect could go structural
- 12–18 months: Google A2UI + Figma MCP integration
- 18–24 months: Anthropic/OpenAI shipping design IDE

If we ship something demonstrable in 3–6 months, the corpus + verification channel are our moat. Architecture diagrams alone are replicable in a quarter.
